The goal of this lab is to develop and integrate distributed applications.

Take some some time to read the whole subject, understand the architecture, etc.

We are going to write service-oriented code where each application focuses on a single business or technical responsibility.

Hipsters call this "micro-services". It’s nothing new…​

Overview

The theme for this lab is that of a chat service whose architecture is as follows:

The chat web application is going to be your main development project. This application will be built in Java using Spring Boot, Maven and the FreeMarker template engine.

This application offers a web interface where one can both:

  1. consult the past messages, and

  2. post new messages, and

  3. post the current posts history to the GitHub Gist service.

The chat application is going to work across replicated instances using the Hazelcast middleware. In practice, this means that all instances of your application across the same network segment will share the very same messages. In practice, this would allow for failover with a front HTTP load balancer.

The chat application only allows for messages to be posted by authenticated users. To do so, you will integrate with a provided authentication service. This is a simple application (written in Go) that exposes a REST/JSON API to check for authentication credentials.

You will also develop a separate Hazelcast-based application that listens for messages to be posted, and append them to a log file.

You will offer a REST/JSON API so that any third-party application can check the messages board.

Finally, you will write a console client for the chat system.

We will not write secure code: authentication messages should at the very least be carried over secure HTTPS exchanges.

Authentication service

Overview

The authentication service is written in Go and fits into a single file:

A pre-compiled binary file may have been given to you by your lab supervisor. Use the source code as a reference or if you need to build it yourself.
$GOPATH/src/auth-service/main.go
package main

import (
  "os"
  "log"
  "net/http"
  "encoding/json"
)

type auth_request struct {
  Login string `json:"login"`
  Password string `json:password`
}

type auth_response struct {
  Success bool `json:"success"`
}

type users struct {
  Users []struct {
    Login string
    Password string
  }
}

func AuthHandler(resp http.ResponseWriter, req *http.Request) {
  log.Println("Authentication request from: ", req.RemoteAddr)
  if (req.Method != "POST") {
    log.Println("Bad authentication request, got a HTTP ", req.Method)
    resp.WriteHeader(http.StatusMethodNotAllowed)
    return
  }
  decoder := json.NewDecoder(req.Body)
  areq := new(auth_request)
  err := decoder.Decode(areq)
  if err != nil {
    log.Println("Bad authentication request: ", err)
  } else {
    response := auth_response{Success: false}
    password, present := Users[areq.Login]
    if present && password == areq.Password {
      response.Success = true
    }
    resp.Header().Set("Content-Type", "application/json")
    encoder := json.NewEncoder(resp)
    err = encoder.Encode(response)
    if err != nil {
      log.Println("Could not encode a JSON response: ", err)
    }
  }
}

var Users map[string]string

func loadConfiguration() {
  file, err := os.Open("users.json")
  if err != nil {
    log.Fatal("Could not open users.json: ", err)
  }
  decoder := json.NewDecoder(file)
  users := new(users)
  err = decoder.Decode(users)
  if err != nil {
    log.Fatal("Could not parse users.json: ", err)
  }
  Users = map[string]string{}
  for _, user := range users.Users {
    Users[user.Login] = user.Password
  }
  log.Println("Users loaded")
}

func main() {
  loadConfiguration()
  http.HandleFunc("/auth", AuthHandler)
  log.Println("Serving requests on port 8666")
  err := http.ListenAndServe(":8666", nil)
  if err != nil {
    log.Fatal("Could not start the server: ", err)
  }
}

The login and passwords are given in a users.json file that must be in the folder from which you run the service. Here is a sample configuration file that defines some users:

users.json
{
  "Users": [
    {
      "Login": "foo",
      "Password": "bar"
    },
    {
      "Login": "Franky",
      "Password": "Vincent"
    },
    {
      "Login": "Louis",
      "Password": "La#Broc4nt3"
    }
  ]
}

The service runs a HTTP service on port 8666. Authentication requests are HTTP POST requests on /auth with JSON requests documents of the form {"login":"…​", "password":"…​"}. The responses are simple `{"success":"true/false"} JSON documents.

Your work

  1. Run the authentication service binary (this requires a valid users.json file!).

  2. Add some login / password entries to users.json. Restart the application

  3. Make some sample authentication request JSON files, and test the service from a shell with curl.

Here is how to do a HTTP POST request with curl, and pass the content of some file sample-request.json as the HTTP POST request body: curl -v -X POST --data-binary @sample-request.json http://localhost:8666/auth.

Spring MVC

Overview

The Spring MVC library is available in our project to make handling HTTP requests easy. In our configuration, it comes with the Freemarker template engine to render HTML documents on the server-side.

A Spring MVC controller is a class annotated with org.springframework.stereotype.Controller.

Each controller method can be annotated with org.springframework.web.bind.annotation.RequestMapping to specify which URL it should handle. The return value of such methods can be:

  1. a String giving the name of the view to render, or

  2. a org.springframework.web.servlet.ModelAndView object that gives the view name as well as values to be used by the template engine.

Message board service

Overview

The message board service is going to be a list of posts. We are going to use Hazelcast to provide replication of the posts across instances of our chat application running on the same network segment: http://hazelcast.org/

Hazelcast is a middleware that provides many group membership services. We are going to focus on one of the simplest feature it provides: the distributed replication of values in a collection. Especially, we can create a list, give it a name, and any modification to the list is replicated:

Config config = new Config();
HazelcastInstance instance = Hazelcast.newHazelcastInstance(config);
IList<Integer> l instance.getList("my-ints");

// This is being replicated magically
l.add(1);
l.add(2);

A message posted to the board will be represented by the following class:

package fr.insalyon.telecom.chat.model;

import java.io.Serializable;

public class Post implements Serializable {

  private String author;
  private String message;

  public Post(String author, String message) {
    this.author = author;
    this.message = message;
  }

  public String getAuthor() {
    return author;
  }

  public String getMessage() {
    return message;
  }

  @Override
  public String toString() {
    return "Post{" +
        "author='" + author + '\'' +
        ", message='" + message + '\'' +
        '}';
  }
}
Why is Post a serializable class?

Your work

Create the following MessageBoard class:

package fr.insalyon.telecom.chat.services;

import com.hazelcast.config.Config;
import com.hazelcast.core.Hazelcast;
import com.hazelcast.core.HazelcastInstance;
import com.hazelcast.core.IList;
import fr.insalyon.telecom.chat.model.Post;
import org.springframework.stereotype.Component;

import java.util.List;

@Component
public class MessageBoard {

  private final HazelcastInstance instance;

  public MessageBoard() {
    Config config = new Config();
    instance = Hazelcast.newHazelcastInstance(config);
  }

  public List<Post> getPosts() {
    return getList();
  }

  private IList<Post> getList() {
    return instance.getList("message-board");
  }

  public void post(Post post) {
    getList().add(0, post);
  }
}

GitHub Gist integration

Overview

GitHub Gist is a popular service to publish text / code snippets: https://gist.github.com/

What we would like now in the chat interface is a button / link to publish all messages to an anonymous Gist. You can think of this as a poor man backup strategy.

Once published, a message shall be posted to the board with the URL to the newly created Gist, e.g.:

mrbean: published the log to https://gist.github.com/lastguest/4373644

Your work

Creating an anonymous Gist is documented here: https://developer.github.com/v3/gists/#create-a-gist

You can take advantage of RestTemplate as we did in the previous code.

The GitHub API JSON documents have tons of entries you don’t really care about as what you really want is the URL. See http://spring.io/guides/gs/consuming-rest/ on how the com.fasterxml.jackson.annotation.JsonIgnoreProperties annotation can be used on your mapping class (@JsonIgnoreProperties(ignoreUnknown = true)).
You have now covered the minimal requirements. Congrats, enjoy, and get ready for more work.

Going further

There are many enhancements to be made, both to polish the application and further your understanding of distributed applications.

Feel-free to cherry pick among the following exercices.

Chat refresh (quite easy)

The chat view is only refreshed when either the user refreshes the page, or when a message is posted.

An easy improvement would be adding a "refresh" link or button on the page.

You could probably perform auto-refresh, too. Think of simple and more elaborated solutions, and implement them.

Logging application (easy)

Create an application that:

  1. uses Hazelcast to join chat board groups, and

  2. monitors additions to post lists, and

  3. appends each new post to a text file.

See http://docs.hazelcast.org/docs/3.3/javadoc/com/hazelcast/core/IList.html and http://docs.hazelcast.org/docs/3.3/javadoc/com/hazelcast/core/ICollection.html#addItemListener(com.hazelcast.core.ItemListener,%20boolean) on monitoring Hazelcast events.

The application is a separate project / code base.

Given that Hazelcast ships as a no-dependency JAR, you can download it from http://hazelcast.org/download/ and just build + run using javac and java.

REST/JSON API (easy)

Make a REST/JSON API to allow any third-party application to see the messages when doing HTTP GET requests on /api/messages.

See the documentation from http://spring.io/guides/gs/rest-service/ on how to create REST controllers.

You can test your API with curl, your new best friend.

Deal with failures (easy)

What happens when the authentication service is down? When GitHub is down?

Make the appropriate changes so that:

  1. the end user sees a friendly error message instead of a technical error, and

  2. the application works in degraded mode if a non-critical service is not available.

Web sessions in Hazelcast (hard)

You can switch to any application instance and enjoy the same messages board.

This is horizontal scaling.

Great, but if you do so then you will still encounter a problem: you need to log into the next instance. You can’t just change instances unless you were logged in each already.

This is failover, but since HTTP sessions are tied to instances, you don’t have transparent failover.
To do

Make clever usage of Hazelcast and cookies manipulation to get global sessions across instances.

Console chat client (easy)

Easy: use Hazelcast to make a separate console chat client that displays the new posts and allows posting new ones. Enjoy System.out and System.in!

Again, Hazelcast is a JAR with no dependencies, so you can directly use it.

Web sockets (hard)

With web sockets, you can make a true chat web application where changes are being pushed to clients.

See http://docs.spring.io/spring/docs/current/spring-framework-reference/html/websocket.html, and introduce web sockets!

This involves some client-side JavaScript wizardry.
